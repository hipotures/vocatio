# ML Boundary Verifier Design

## Goal

Build a separate machine-learning verifier pipeline for image-only candidate boundaries before attempting Gemma 4 tuning. The new verifier will classify candidate gaps generated by the existing heuristic pipeline and predict two labels:

- `segment_type`: `performance`, `ceremony`, or `warmup`
- `boundary`: `true` or `false`

The verifier is intended to improve false positive and false negative decisions at candidate gaps without replacing the current candidate generator. If the ML verifier is not good enough, the same candidate dataset will later be reused for Gemma 4 E4B tuning.

## Scope

This design covers:

- candidate dataset generation from existing per-day artifacts and final reviewed segment truth
- feature extraction for a tabular verifier
- an optional AutoMM experiment that adds thumbnail images to the same candidate records
- training and evaluation workflows based on AutoGluon
- production usage as an alternative verifier for the same candidate gaps

This design does not cover:

- replacing the candidate generator
- replacing the review GUI
- Gemma 4 fine-tuning implementation
- adding VLM outputs as training features in the first iteration

## Existing Context

The repository already has a mature image-only pipeline that provides:

- canonical per-day media export via `export_media.py`
- photo proxy extraction via `extract_embedded_photo_jpg.py`
- per-photo quality annotations
- DINOv2 embeddings and boundary feature generation
- heuristic boundary bootstrap and segment generation
- review GUI and reviewed segment outputs
- optional per-photo annotation JSON and optional VLM-based boundary probing

The current image-only architecture uses a candidate-generator style flow already:

- candidate gaps are selected by time-gap heuristics
- a verifier layer then decides whether a candidate is a true split

The new ML verifier should reuse that structure rather than introducing a full sequential model over all photos.

## Design Principles

- Keep the current candidate-generation architecture.
- Train on the same kinds of candidate records the production system will see.
- Separate candidate-generator quality from verifier quality.
- Make the candidate dataset reusable for later Gemma 4 tuning.
- Keep v1 narrow and diagnosable.
- Use AutoGluon for rapid iteration and strong tabular baselines.
- Treat image input as an optional experiment, not a required dependency for the first working system.

## Labels and Truth Model

The source of truth is the final manually corrected segmentation for a day.

Each photo in reviewed truth must be attributable to:

- a `final_segment_id`
- a `final_segment_type`

Supported segment types in v1:

- `performance`
- `ceremony`
- `warmup`

Semantics:

- one performance is one segment
- one ceremony block is one segment
- one warmup block is one segment

No further subdivision of ceremony or warmup is attempted in v1.

For each candidate gap, the pipeline derives:

- `boundary = true` if the photos on the two sides of the candidate belong to different final segment ids
- `boundary = false` otherwise

`segment_type` for a candidate record is defined as the type of the segment on the right side of the candidate. This is the canonical convention for candidate records and must be used consistently in training and evaluation.

## Candidate Definition

A candidate is not an arbitrary photo pair and not a sliding window over every photo.

A candidate is a gap location produced by the same heuristic candidate generator used in production, based primarily on time-gap rules.

This is intentional:

- the verifier should be trained on the same input distribution it will see at inference time
- easy non-candidate interior windows should not dominate the dataset
- the system's total recall remains partly limited by candidate generation, and this should be measured separately rather than hidden inside verifier metrics

Each candidate record contains a fixed local window around the candidate center. The initial standard window is 5 photos, with the boundary decision defined between frame 3 and frame 4 of the window.

The window definition is fixed and must not vary across records in v1.

## Dataset Record Shape

Each candidate record should include at least:

- `day_id`
- `candidate_id`
- `window_size`
- `center_left_photo_id`
- `center_right_photo_id`
- `window_photo_ids`
- `window_relative_paths`
- `left_segment_id`
- `right_segment_id`
- `segment_type`
- `boundary`
- `center_gap_seconds`
- `was_user_corrected`
- `is_hard_negative`
- `split_name`

Records should be stored in a tabular format suitable for AutoGluon, ideally Parquet plus an explicit manifest CSV or JSON for diagnostics.

## Feature Groups

### 1. Time and sequence features

These are derived from timestamps and local gap structure around the candidate.

Examples:

- `center_gap_seconds`
- `left_gap_mean`
- `right_gap_mean`
- `local_gap_median`
- `gap_ratio`
- `gap_is_local_outlier`
- `max_gap_in_window`
- `gap_variance`

Raw gap information is allowed in the tabular verifier because this model is explicitly intended to use structured features. This is different from the VLM prompt, where gap text previously caused shortcut behavior.

### 2. Descriptor-difference features

These are derived from existing per-photo annotation JSON, but only as stable structured signals. Raw natural-language descriptions are not used.

Examples:

- left/right dominant-color agreement
- left/right costume-type agreement
- left/right visible-person-count bucket change
- left/right garment-category change
- descriptor consistency score within left half of window
- descriptor consistency score within right half of window
- left/right group-change flags

`visible_person_count` must be treated as a weak feature only. It cannot be interpreted as a direct proxy for solo versus group because group performances often produce single-person frames.

### 3. Candidate metadata features

Examples:

- candidate rank within day
- whether candidate was selected only by time-gap rule or also by a secondary heuristic if such a rule exists later
- whether the candidate was explicitly corrected by the user during review

## Experimental Variants

### Variant A: Tabular-only baseline

This is the required first implementation.

Input:

- candidate-level tabular features only

Modeling:

- AutoGluon tabular predictors
- multi-label prediction implemented as ordered predictors
- recommended order:
  1. `segment_type`
  2. `boundary`, conditioned on predicted or true `segment_type`

This is the production candidate for v1.

### Variant B: Tabular + thumbnail experiment

This is the optional experimental branch that should still be implemented in the same project because it uses the same candidate records and evaluation logic.

Input:

- the same tabular features as Variant A
- candidate thumbnail images derived from the existing `thumb` proxies

This should be implemented using AutoMM in AutoGluon. The goal is not to make image input mandatory, but to cheaply test whether small visual context improves over the tabular baseline.

This variant must use the same splits and report format as the tabular baseline.

### Not included in v1

The following are intentionally excluded from the first verifier project:

- current VLM outputs as training features
- preview images in the first benchmark matrix
- full-resolution images
- end-to-end sequence models over all photos

## Data Splits

Splits must be done by whole days or whole events, never by random candidate rows.

This avoids leakage where near-identical sequences from the same event appear in both training and evaluation.

Recommended split structure:

- train: the majority of available days
- validation: a separate set of full days
- test: a separate set of full days, ideally including different years, different cameras, or visibly different visual conditions

The system should support explicit split manifests rather than hidden random splits.

## Metrics

Evaluation must be reported at two levels.

### 1. Verifier-level metrics

Per candidate:

- boundary precision
- boundary recall
- boundary F1
- confusion matrix
- segment-type accuracy or macro F1

These must also be broken out by:

- `performance`
- `ceremony`
- `warmup`

### 2. Reconstructed segment metrics

After applying candidate predictions back to a day:

- false split count
- missed split count
- merge error count
- number of predicted segments versus reviewed segments

This second level is the most important operational score because it reflects how much manual cleanup the user still needs to do in the GUI.

### Candidate-generator diagnostics

These are separate from verifier metrics and must not be blended into model-quality metrics:

- fraction of true boundaries that became candidates
- count of reviewed boundaries never surfaced to the verifier

## Production Usage

The ML verifier should initially be added as an alternative verifier for the same candidate gaps already used in the image-only pipeline.

That means:

- candidate generation remains unchanged
- the ML verifier runs on the same candidate set
- outputs can be compared directly with the current heuristic or VLM-based verifier

The first production goal is not immediate replacement. The first goal is side-by-side comparison and reliable offline evaluation.

If the ML verifier outperforms the existing verifier on operational metrics, it can later become the primary verifier for image-only candidate gaps.

## New CLI Components

The project should add four main scripts.

### `build_ml_boundary_candidate_dataset.py`

Responsibilities:

- load per-day reviewed truth
- run the candidate generator over day data
- construct candidate windows
- derive labels and candidate metadata
- write candidate-level dataset artifacts

### `validate_ml_boundary_dataset.py`

Responsibilities:

- verify path integrity
- verify split integrity
- verify label completeness
- verify window consistency
- report class balance and hard-negative coverage

### `train_ml_boundary_verifier.py`

Responsibilities:

- run AutoGluon tabular baseline
- optionally run AutoMM thumbnail experiment
- save models, metadata, feature config, and training summaries

### `evaluate_ml_boundary_verifier.py`

Responsibilities:

- evaluate saved models on validation or test split
- compute verifier-level metrics
- reconstruct per-day segment outputs
- compute operational metrics
- write comparison reports

## Dataset Reuse for Gemma 4

The candidate dataset produced here must be structured so it can later be reused for Gemma 4 tuning.

That means each candidate record must preserve:

- ordered image paths for the candidate window
- canonical labels for `segment_type` and `boundary`
- enough provenance to reconstruct the original day and photo order

The ML project is therefore not throwaway work. It is the first supervised-labeling stage for a later multimodal fine-tuning path.

## Error Handling and Failure Modes

The pipeline should fail open at the day level where possible and never silently drop candidate records.

Candidate rows with missing optional descriptors may still be kept if required core fields are present.

Rows with missing required window images or inconsistent ordering should be surfaced explicitly by validation and excluded from training with a recorded reason.

Training and evaluation reports must make it obvious whether poor results came from:

- missing candidate coverage
- weak tabular features
- weak thumbnail signal
- leakage or split problems
- severe class imbalance

## Testing Strategy

Testing should include:

- unit tests for candidate-label derivation
- unit tests for feature extraction from descriptor JSON
- unit tests for split integrity checks
- integration tests on small synthetic per-day fixtures
- script `--help` coverage and representative local invocations

A regression fixture should include at least:

- a true performance boundary
- a long-gap no-boundary hard negative within the same performance
- a ceremony block with strong internal visual variation but no internal boundary
- a warmup block with no internal boundary

## Recommended v1 Benchmark Matrix

The initial benchmark matrix should be deliberately small:

- `tabular_only`
- `tabular_plus_thumbnail`

Both should use:

- the same candidate dataset
- the same split manifests
- the same label definitions
- the same report schema

If tabular-only is already clearly strong, the image experiment is still useful as a decision point before Gemma tuning.

## Out of Scope Decisions Explicitly Deferred

The following decisions are postponed until after v1 evaluation:

- whether to use preview images instead of thumbnails in AutoMM
- whether to add VLM outputs as features
- whether to build an ensemble combining ML verifier and VLM verifier
- whether to promote the ML verifier to the production default
- whether to proceed to Gemma 4 tuning
